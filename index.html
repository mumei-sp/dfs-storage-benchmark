Test Setup: 1,000 Pre-Seeded Objects

LIST benchmarks run against 1,000 zero-byte objects under prefix perf-list-seed/obj-00000 → obj-00999. Each LIST call returns all 1,000 objects in a single page (maxKeys=1000). Object size tiers (tiny→xlarge) don't affect LIST — it's purely metadata.

Observation: Even with only 1,000 objects, RustFs only manages 4-7 ops/s — the metacache doesn't eliminate the overhead of cross-drive directory walking. The benchmark measures this specific scale; results at larger object counts were not tested.

💡 Optimization Tip — Reduce LIST Scope for RustFs: Since RustFs walks filesystem directories, you can reduce the scan scope by:
• Prefix partitioning: Use hierarchical prefixes (e.g., /photos/2024/01/ instead of flat /photos/) — RustFs only walks the subdirectory matching the prefix
• Multiple buckets: Split objects across buckets by tenant/category — each bucket is a separate directory tree, so listing one bucket doesn't scan others
• Metacache warmth: Repeated LISTs on the same prefix are served from RustFs's S2-compressed metacache — the first call is slow, subsequent calls are faster
For Ozone and SeaweedFS, these optimizations provide minimal benefit — their LSM-tree sorted key seek is already O(log N) regardless of how objects are organized.

Why the 60-75x Gap? Three Architectures, Three LIST Strategies

RustFs (MinIO-style)

LIST request arrives at any node
  → walk filesystem dirs across ALL drives in ALL sets
  → readdir() syscalls for each directory
  → filter results by prefix
  → sort and paginate
  → return max 1000 objects
  • O(N) initial walk First LIST walks filesystem dirs via readdir() syscalls across all drives
  • No persistent index Filesystem directory structure IS the only source of truth
  • Metacache Results cached as S2-compressed blocks in .minio.sys — subsequent LISTs stream from cache
  • Cold LIST = slow Cache miss → full directory walk; cache hit → fast. Benchmark measures cold performance
  • Fan-out Must query multiple nodes (objects hash-distributed across erasure sets)

Apache Ozone

LIST request → S3 Gateway → OM lookupKey
  → RocksDB iterator.seek(prefix)
  → sequential forward scan (sorted keys)
  → stop at maxKeys or prefix mismatch
  → return page + continuation token
  • O(log N + K) Seek to prefix, scan K results — independent of total object count
  • RocksDB Sorted LSM tree — keys with same prefix are physically adjacent
  • Block cache Hot SST blocks stay in memory — zero disk I/O for repeated scans
  • Concurrent Each request gets its own RocksDB iterator (lock-free MVCC)
  • Single node OM has the complete index — no fan-out needed

SeaweedFS

LIST request → S3 Gateway → Filer
  → LevelDB iterator.seek(parent_dir + prefix)
  → sequential forward scan (sorted keys)
  → stop at maxKeys or prefix mismatch
  → return page (no volume server access)
  • O(log N + K) Same LSM-tree seek as Ozone — keys sorted by (parent_dir, filename)
  • LevelDB Embedded LSM tree — same family as RocksDB but lighter/simpler
  • No volume access LIST is purely a Filer metadata operation — volume servers never touched
  • Direct local read Filer reads its own LevelDB — no RPC to another process (unlike Ozone S3G→OM)
  • Single process S3 gateway + Filer in same process (weed filer -s3) — zero network hop for metadata

Both Ozone and SeaweedFS use embedded LSM-tree databases for metadata (RocksDB vs LevelDB). So why does SeaweedFS win by 1.0-1.4x on LIST?

SeaweedFS Edge Over Ozone on Medium/Large LIST (1.3-1.4x)

While both systems use LSM-tree metadata stores and perform similarly on tiny/small LIST (1.0x), SeaweedFS pulls ahead on medium and large sizes (1.3-1.4x). Three architectural factors explain the gap:

  • LevelDB is simpler/lighter than RocksDB — fewer features means less CPU overhead per seek. RocksDB supports column families, bloom filters, multiple compaction styles, and compression codecs. LevelDB is a stripped-down LSM with one compaction strategy. For a simple range scan returning 1,000 keys, LevelDB's simplicity wins.
  • Filer is a single Go process vs Ozone's S3G→OM RPC chain — SeaweedFS S3 gateway and Filer run in the same process (weed filer -s3). The LIST request never leaves the process boundary to reach LevelDB. Ozone's LIST crosses two process boundaries: S3G (HTTP) → OM (Ratis RPC) → RocksDB. Each hop adds serialization, context switching, and network latency.
  • No Raft overhead on the metadata read path — SeaweedFS Filer reads its local LevelDB directly with zero consensus overhead. Ozone's OM serves reads through the Ratis state machine — even read-only operations go through the Raft leader, adding dispatch overhead. With 1,000 objects per response, this per-request overhead accumulates.

Note: This gap may narrow at larger object counts or with Ozone's RocksDB block cache fully warmed. The benchmark tests 1,000 objects — at 100K+ objects, RocksDB's bloom filters and advanced caching may close the gap.

Write Path Architecture

RustFs Write Path

HTTP PUT → hash(key) → select erasure Set
→ Reed-Solomon encode (1MiB blocks)
→ parallel write shards to drives in Set
→ write xl.meta atomically per drive
→ quorum ACK (data+1 drives) → fdatasync
→ return 200
  • No coordination No Raft, no openKey/commitKey round-trips
  • Parallel shards All erasure shards written simultaneously
  • 1 network hop Client → any node, shards to local drives
  • 0% failures No Raft pipeline to stall or timeout
  • EC overhead Reed-Solomon encoding uses CPU

Ozone Write Path (EC RS-6-3)

HTTP PUT → S3 Gateway
→ OM openKey (Ratis WAL commit)
→ SCM allocate EC block group (9 blocks across DNs)
→ ECKeyOutputStream: RS 6+3 encode (1024KB stripe)
→ parallel write 9 chunks across 3 datanodes
  (3 chunks per DN × 3 racks)
→ OM commitKey (Ratis WAL commit)
→ return 200
  • 2 Raft round-trips openKey + commitKey through OM Raft group (all OM mutations go through Ratis when HA enabled)
  • Parallel EC writes All 9 chunks (6 data + 3 parity) written simultaneously — NOT sequential Raft chain
  • 4+ network hops Client → S3G → OM → ECKeyOutputStream → 3 DNs → OM
  • 7-34% failures EC stripe coordination failures under load — partial writes require cleanup
  • EC overhead RS 6+3 encoding on client-side (S3G) + 1024KB stripe buffering
  • 9 block locations SCM must allocate 9 blocks across 3 racks — more allocation overhead than RATIS/3

SeaweedFS Write Path (Replication)

HTTP PUT → S3 Gateway (weed s3)
→ Filer: create entry in metadata store (LevelDB)
→ Master: assign(volumeId, fileKey, cookie)
→ POST data directly to volume server
  → write needle to .dat file (append-only)
  → update in-memory needle map (16 bytes/entry)
  → CRC32-C checksum stored with needle
  → synchronous replicate to N volume peers
→ Filer: store chunk→volume mapping
→ return 200
  • Lightweight coordination Master only assigns volume ID — no per-write consensus
  • Append-only writes Needle appended to volume .dat file — sequential I/O, no random writes
  • 40 bytes overhead/file Minimal metadata per needle (Facebook Haystack design)
  • 0% failures No EC coordination, simple replication — 100% success at all concurrencies
  • 2 metadata hops Filer metadata write + Master volume assignment (both lightweight)

Ozone Read Path (EC RS-6-3)

HTTP GET → S3 Gateway → OM lookupKey
→ get EC block group locations (9 containers)
→ ECKeyInputStream: read 6 data chunks sequentially
  from 6 block locations across 3 datanodes
  (skip 3 parity unless data chunk unavailable)
  (parallel reads used in degraded/reconstruction mode)

Why SeaweedFS InputStream Crashes to 80 ops/s (vs 20K Path) — And Why Ozone Doesn't

SeaweedFS Path mode delivers 20,203 ops/s at c64 — but InputStream drops to just 80 ops/s (a 250x collapse). Ozone only loses 12% in InputStream mode. The difference is in how each system's middleware handles streaming reads.

Ozone InputStream: Direct DN Streams (5,915 → 5,230 ops/s, -12%)

Client sends GET → S3G
  │
  ├─ S3G asks OM: "where is object X?"
  │  OM returns: [DN1:chunk1, DN2:chunk2, DN3:chunk3]
  │
  ├─ S3G opens gRPC streams DIRECTLY to DataNodes
  │  ECKeyInputStream lives INSIDE S3G process
  │
  │  DN1 ──gRPC stream──→ ┐
  │  DN2 ──gRPC stream──→ ├─ ECKeyInputStream (in S3G)
  │  DN3 ──gRPC stream──→ ┘      │
  │                          EC decode
  │                               │
  └─ S3G streams HTTP response ←──┘
     to Client
  • Direct DN connection S3G opens persistent gRPC streams to DataNodes — data flows continuously, no per-chunk HTTP round-trips
  • OM not in data path OM only returns block locations (metadata). It never sees or buffers actual object data
  • EC decode is CPU-only Chunks arrive via stream, EC reassembly happens in-process — no extra network hops
  • 12% drop InputStream adds JVM byte[] allocation + GC pressure vs Path's zero-copy sendfile() — modest overhead

SeaweedFS InputStream: Filer Proxy Bottleneck (20,203 → 80 ops/s, -99.6%)

Client sends GET → S3 Gateway → Filer
  │
  ├─ Filer looks up LevelDB: path → [vol:3, needle:01637037d6]
  │
  ├─ Filer makes HTTP request TO Volume Server
  │  Volume Server reads needle → sends bytes back
  │
  │  Volume Server ──HTTP──→ Filer (Go memory buffer)
  │                            │
  │                    data BUFFERED in Filer RAM
  │                            │
  └─ Filer streams buffered ←──┘
     data back to Client

The problem: For chunked/streaming reads, the Filer makes a separate HTTP request to the Volume Server for each chunk. The data is copied: Volume disk → Volume memory → Network → Filer memory buffer → Network → Client. Two full network transfers per chunk.

  • Filer proxies ALL bytes Every byte of object data flows through Filer's Go memory — Filer is not just metadata, it's a full data proxy
  • Per-chunk HTTP round-trip For files split into 4MB chunks, each chunk = new HTTP request from Filer → Volume Server. A 3MB file may be 1 chunk, but the HTTP overhead per request compounds at c64
  • Range-read bug (#7056) Confirmed: range requests caused Filer to read the entire file from Volume Server, not just the requested range — 1000x bandwidth amplification in production
  • Context cancellation (#7395) Confirmed: streaming context was coupled to HTTP request context — slow client reads (even a JVM GC pause) cause Filer to abort the Volume Server connection. ~75% failure rate on large files
  • 64 concurrent proxy sessions At c64, Filer juggles 64 simultaneous proxy sessions — each with its own Volume Server connection + memory buffer + client connection. Filer's CPU, RAM, and network all saturate

Bottom line: Ozone's S3G acts as a thin routing layer — it gets locations from OM, then streams data directly from DataNodes through the EC decoder. The data path is: DN → gRPC → S3G (stream-through) → Client. SeaweedFS's Filer acts as a full data proxy — it fetches data from Volume Servers into its own memory, then re-transmits to the client. This double-hop + buffering + per-chunk HTTP overhead is what turns 20K ops/s into 80. Path mode avoids this because the entire object is fetched and returned in a single HTTP response — the proxy overhead is amortized over one request instead of many chunks.

Why SeaweedFS Dominates HEAD (24K vs Ozone 10K vs RustFs 1.6K)

HEAD is a pure metadata operation — no object data is transferred. The performance is entirely determined by how fast each system can look up and return metadata:

  • RustFs (1,585 ops/s): Must read xl.meta from all 8 drives in the erasure set, quorum vote, then return headers. At c64, that's 512 disk I/O ops competing for SATA SSD bandwidth → saturates at ~1,600 ops/s.
  • Ozone (10,188 ops/s): Single RocksDB get() in OM, served from block cache (in-memory). Fast, but requires S3G → OM RPC round-trip + per-request OzoneClient allocation.
  • SeaweedFS (24,353 ops/s): Single LevelDB get() in Filer, served in-process (no RPC). The S3 gateway and Filer share the same Go process — HEAD resolves in a single function call to LevelDB. No network hop, no client allocation, no serialization overhead. This is why SW is 2.4x faster than Ozone despite both using LSM-tree metadata stores.

⚠ Ozone DELETE — The Deferred Cost

Ozone's 3x DELETE speed advantage is partly borrowed from the future. The benchmark measures time-to-ACK, but Ozone defers the actual work:

DELETE request → OM deleteKey()
  Step 1: Remove key from keyTable (RocksDB)     ← benchmarked (fast)
  Step 2: Add key to deletedTable (RocksDB)       ← benchmarked (fast)
  Step 3: Return 200 OK to client                 ← benchmark stops here
  ──────────────────────────────────────────────
  Step 4: KeyDeletingService scans deletedTable    ← background, every ~60s
  Step 5: Send block delete commands to SCM        ← background
  Step 6: SCM instructs datanodes to reclaim       ← background
  Step 7: Datanodes delete actual EC chunks/blocks ← background, disk I/O
  Step 8: Disk space finally freed                 ← minutes later

Production side-effects of deferred deletion:

GC Storm Bulk deletes degrade concurrent I/O

When you delete 1 million objects, all DELETEs return 200 OK in seconds. But the background now has a problem:

KeyDeletingService wakes up (every ~60s)
  → finds 1,000,000 entries in deletedTable
  → sends 1,000,000 block-delete RPCs to SCM
    → SCM sends commands to 3 datanodes per block
      → each DN does actual disk I/O to delete EC chunks

Meanwhile, your normal traffic is also running:
  PUT (needs SCM to allocate blocks)  ← SCM busy processing deletes
  GET (needs DN disk I/O)             ← DNs busy deleting blocks
  LIST (needs OM RocksDB)             ← OM compacting deletedTable

All three components (OM, SCM, DNs) are doing delete cleanup on top of serving live traffic. PUT latency spikes because SCM is processing delete commands instead of allocating blocks. GET latency spikes because datanodes are doing delete I/O on the same disks serving reads.

RustFs: No central SCM — each node cleans its own trash independently, so delete GC doesn't create a single-coordinator bottleneck.

deletedTable Bloat High-churn workloads grow OM memory

Workloads that constantly create and delete objects (e.g., Spark shuffle files, temp processing artifacts) cause the deletedTable to grow:

Every hour: write 100K temp objects → delete 100K temp objects
  keys move: keyTable → deletedTable → (GC cleans eventually)

If GC can't keep up with churn rate:
  Hour 1:  deletedTable: 100K entries (GC cleaned 80K, 20K pending)
  Hour 2:  deletedTable: 120K entries (backlog growing)
  Hour 5:  deletedTable: 300K entries
  Hour 24: deletedTable: millions of entries

This matters because:

  • deletedTable lives in OM's RocksDB — more entries = more memory for block cache, more SST files, more compaction CPU
  • KeyDeletingService scans the table every ~60s — scanning millions of rows burns OM CPU constantly
  • RocksDB compaction triggered by the growing table causes periodic latency spikes on ALL OM operations (LIST, HEAD, lookupKey)

Essentially a metadata memory leak — the data is gone from the user's perspective, but cleanup bookkeeping accumulates and degrades OM performance.

  • Disk-full alerts keep firing during the GC lag window even though you've already deleted data
  • Auto-scaling based on disk usage won't trigger correctly — it sees "full" disk until GC catches up
  • Capacity planning must account for this: effective free space = reported free + pending deletions in deletedTable

Similar disk-space lag, but RustFs has no central coordinator (SCM) that can become a bottleneck during GC storms. Each node cleans its own trash independently.

Step 1 How the Double Buffer works (HDDS-1512)

Scalability Bottleneck — The Central Metadata Problem

Ozone's Achilles Heel: OM Becomes the Bottleneck at Scale

Note: Our test uses a single OM (no Raft HA). The Raft overhead described below applies to production Ozone deployments where OM HA is enabled (3 or 5 OMs). In our test, OM overhead is from RocksDB + double-buffer only — no Raft consensus. Production would be slower for writes due to added Raft round-trips.

Every single Ozone operation — PUT, GET, HEAD, LIST, DELETE — must pass through the Ozone Manager (OM). In production, OM is replicated via Raft for HA. This creates a fundamental scaling ceiling:

Ozone at scale — ALL traffic funnels through OM:

  IoT sensors (10K writes/s) ──→ ┐
  Log pipeline (50K writes/s) ──→ ├──→ S3 Gateway ──→ OM (single Raft group)
  ML training (5K reads/s) ─────→ ├                    │
  Analytics (2K LIST/s) ────────→ ┘                    ├→ RocksDB (one instance)
                                                       ├→ Raft WAL (serial writes)
                                                       └→ Table cache (bounded memory)

  Total: ~67K metadata ops/s hitting ONE OM leader node

  OM Leader must:
    • Serialize all writes through Raft consensus (inherently serial)
    • Run validateAndUpdateCache for every transaction
    • Maintain table cache in memory
    • Flush double-buffer to RocksDB (compaction competes with reads)
    • Serve all lookupKey reads from cache + RocksDB
    • Handle Raft heartbeats and follower replication

    replication overhead, not more throughput.

    namespace across multiple OMs. But adds
    routing complexity and cross-OM operations

  Throughput is limited by network and EC encoding,

Overwrite Storm 0% Ozone — Concurrent writers silently rejected

Ozone uses optimistic locking for overwrites. When two writers try to update the same key simultaneously:

Writer A: openKey("config.json", expectedGen=5) → granted, gen=6
Writer B: openKey("config.json", expectedGen=5) → granted, gen=7

Writer A: commitKey(gen=6, data="version-A") → SUCCESS ✓
Writer B: commitKey(gen=7, data="version-B")
  → OM checks: current gen is now 6 (from Writer A)
  → Writer B expected gen=5, but it's 6
  → REJECTED: KEY_NOT_FOUND or generation mismatch

Result: Only Writer A's data persists. Writer B's write is silently lost.
No error returned to Writer B in some code paths → silent data loss.

Real-world scenarios where this breaks:

  • Shared config files: Two deployers updating the same config → one update silently lost
  • Collaborative editing: Two users saving the same document → last-writer-wins expected, but second writer's save is rejected entirely
  • Counter/state objects: Multiple workers updating a progress tracker → updates dropped under contention
  • Log aggregation: Multiple processes appending to the same log object → only one succeeds per cycle

💡 Mitigation strategies:
• Use unique keys: Instead of overwriting status.json, write status-{timestamp}.json — avoids contention entirely
• CAS retry loop: Read current generation → write with expected generation → retry on conflict (like database optimistic locking)
• External locking: Use ZooKeeper or a distributed lock before writing shared mutable objects
• RustFs for mutable objects: RustFs's atomic xl.meta replace handles concurrent overwrites with ~93% success (last-writer-wins semantics)

RustFs 100% RAW How it achieves strong consistency — and caveats

RustFs achieves 100% read-after-write because the read path is deterministic:

PUT "photo.jpg":
  hash("photo.jpg") → erasure Set #7 (drives 3,5,8,11 data + 1,4,9,12 parity)
  → write shards to drives 3,5,8,11,1,4,9,12
  → fdatasync each shard (data physically on disk platter)
  → quorum ACK (need data+1 drives to confirm)
  → return 200 OK

GET "photo.jpg" (immediately after):
  hash("photo.jpg") → erasure Set #7 (SAME drives: 3,5,8,11,1,4,9,12)
  → read from drives 3,5,8,11 (data shards)
  → data IS there — fdatasync guaranteed it before PUT returned
  → return data ✓

The key insight: no intermediate metadata layer. The hash function maps directly to physical drives, and fdatasync ensures data is on those drives before ACK. There's nothing that needs to "catch up".

Caveats and edge cases:

  • Filesystem requirement This only works correctly on XFS or ZFS. On ext4, O_DIRECT + fdatasync semantics can break — data may not be durable even after fdatasync returns
  • Network partitions If a node becomes unreachable after PUT ACK but before the client's GET, and the GET routes to a different node via the load balancer, the new node may not have the data (it was written to the original node's local drives). The Nginx least_conn LB doesn't guarantee same-node routing.
  • Quorum edge case PUT ACKs after quorum (data+1 drives), not ALL drives. If the GET hits a drive that was in the minority (didn't ACK yet), it reads stale xl.meta. RustFs handles this with quorum reads (reads from multiple drives, picks latest version), but this adds latency.

ofPath vs InputStream — How Data Flows Through The JVM

ofPath (Zero-Copy)

Client PUT:
  File on disk
    ↓ OS sendfile() / FileChannel.transferTo()
    ↓ kernel space → NIC directly
    ↓ (data NEVER enters JVM heap)
    → Network → Storage backend

Client GET:
  Storage backend → Network
    ↓ FileChannel.transferFrom()
    ↓ NIC → kernel space → File on disk
    ↓ (data NEVER enters JVM heap)
    → File written
  • Zero heap allocation Data stays in kernel buffers — JVM heap untouched
  • No GC pressure Nothing allocated = nothing to garbage collect
  • No byte[] copies OS handles transfer directly (DMA where available)
  • Requires file Data must be on disk — can't stream from memory

InputStream (Traditional I/O)

Client PUT:
  File / byte source
    ↓ InputStream.read(byte[4MB])
    ↓ copies 4MB chunk into JVM HEAP
    ↓ HttpClient writes byte[] to socket
    ↓ (each chunk: allocate → copy → send → GC)
    → Network → Storage backend

Client GET:
  Storage backend → Network
    ↓ InputStream.read(byte[4MB])
    ↓ copies into JVM HEAP byte[]
    ↓ write byte[] to OutputStream / File
    ↓ (each chunk: allocate → copy → write → GC)
    → File written
  • Heap allocation per chunk 4MB byte[] allocated for every chunk read/write
  • GC pressure Short-lived byte[] arrays → frequent young-gen GC
  • 2 copies per chunk disk → heap → socket (PUT) or socket → heap → disk (GET)
  • Flexible Works with any data source — memory, streams, transformations

Why Does This Matter?

When ofPath Wins Big

  • Large files (64MB): InputStream allocates 16× 4MB byte[] per object → massive GC pressure. Peak heap 1,776MB (IS) vs 593MB (Path) for RustFs PUT large c16
  • Long-running workloads: Soak test: 10,446 GC events (IS) vs 3,990 (Path) for RustFs — 2.6x more GC pauses over 1 hour
  • High concurrency reads: Ozone GET c64: ofPath 5,915 ops/s vs InputStream 5,230 ops/s — 13% faster because less GC interference
  • Tail latency: PUT small c64: stdDev 241ms (Path) vs 345ms (IS) for RustFs — InputStream has more latency jitter from GC pauses

When InputStream Is Acceptable

  • Small files, low concurrency: PUT small c1: almost identical ops/s. GC overhead is negligible when volume is low
  • Metadata operations: HEAD and LIST don't transfer data — I/O path doesn't matter for metadata-only ops
  • When data is in-memory: If data comes from a transformation pipeline (e.g., compression, encryption), you already have byte[] — ofPath adds a disk write
  • Short benchmarks: GC differences only compound over time. A 60-second test barely shows the gap

ofPath: Steady State

Disk Speed Impact Would better disks improve RustFs LIST?

Measured: RustFs LIST uses only 0.6-1.6 MB/s peak disk I/O despite having a 7,093 MB/s read disk. The bottleneck is NOT sequential disk throughput — it's metadata read latency.

How EC Affects Read & Write Paths

Write Path: EC Encoding

RustFs PUT (3MB):
  hash(key) → erasure Set
  → RS encode: 3MB → 4×750KB data + 4×750KB parity
  → parallel write 8 shards to local drives
  → fdatasync each shard
  → quorum ACK → return 200
  Total disk I/O: 6 MB    Network: 0 (local drives)

Ozone PUT (3MB):
  S3G → OM openKey [Raft WAL]
  → RS encode: 3MB → 6×500KB data + 3×500KB parity
  → parallel write 9 chunks to 3 datanodes
    (3 chunks per DN, rack-aware placement)
  → OM commitKey [Raft WAL]
  → return 200
  Total disk I/O: 4.5 MB  Network: 3 MB (cross-node)
  • RustFs writes 33% more bytes (6MB vs 4.5MB) but all local disk
  • Ozone writes fewer bytes but needs cross-node network I/O
  • Ozone also pays 2 Raft round-trips (~10-20ms) for metadata
  • Net result: RustFs 20-35% faster at low-medium concurrency

Read Path: EC Decoding

RustFs GET (3MB):
  hash(key) → erasure Set
  → read 4 data shards (750KB each) from local drives
    (O_DIRECT: bypasses page cache → always hits disk)
  → HighwayHash256 verify each shard
  → RS decode (if degraded) → stream 3MB to client
  Disk reads: 4    Total I/O: 3 MB

Ozone GET (3MB):
  S3G → OM lookupKey [~3-5ms]
  → read 6 data chunks (500KB each) sequentially from 3 datanodes
    (page cache: hot chunks served from RAM)
  → CRC32 verify each chunk
  → reassemble → stream 3MB to client
  Disk reads: 0-6  Total I/O: 3 MB (often from RAM)
  • Both read 3MB total — same bytes over the wire
  • RustFs: 4 reads × always disk = predictable but unscalable
  • Ozone: 6 reads × often RAM = higher base cost, but scales with cache
  • At c64: RustFs 256 disk reads saturate drives → 1,197 ops/s plateau
  • At c64: Ozone cache hits → 5,915 ops/s and still climbing

Bottom Line

2 GB/s is achievable in bandwidth — but server-side CPU is the unknown.

The benchmark was network-bound at 1 Gbps. Both systems saturated the 125 MB/s link at 112 MB/s (89.4% efficiency). Network and disk have 5-70× headroom on enterprise hardware. The question is whether server-side CPU can keep up.

  • What we measured: Client-side SDK CPU is low for both systems (~2% for PUT). Network is the only confirmed bottleneck. Both achieve identical throughput at saturation (~112 MB/s).
  • What we didn't measure: Server-side CPU (EC encoding, Raft consensus, RocksDB). This is the critical unknown for 2 GB/s projections.
  • RustFs architectural advantage: Single Go binary per node, ISA-L SIMD for EC, no metadata coordination, no JVM GC. With 3-4 nodes for EC distribution and 100% measured reliability — architecturally simpler path to 2 GB/s.
  • Ozone architectural concern: 3-4 separate JVMs per node (S3G + OM + SCM + DN), Raft consensus on every write, RocksDB compaction. Requires 7-13 nodes (including infrastructure). OM becomes a bottleneck for small objects (<100 KB at 2 GB/s = 20K+ ops/s).
  • Reliability at scale: RustFs showed 100% success at all measured loads. Ozone showed 80% at sustained c16 on 1 Gbps. Whether this gap persists at 2 GB/s is unknown.

To validate: A benchmark on enterprise hardware (100 GbE + NVMe) with server-side monitoring (CPU, memory, disk per node) is needed. The current results confirm bandwidth is not the limiting factor — but cannot confirm whether server-side resources are sufficient at 18× higher throughput.
